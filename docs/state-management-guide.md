# 状态管理规范草稿

## 文档目的
这份文档用于说明当前前端状态管理的边界、推荐写法和后续扩展方向，避免出现状态放置混乱、重复缓存和数据来源不清晰的问题。

## 当前方案
当前项目使用 Pinia 作为全局状态管理方案。

当前已落地的核心状态：
- token
- 当前用户资料
- 用户角色
- 登录中状态
- 刷新用户信息状态

## 状态分类建议

### 一类：全局共享状态
适合放进 Pinia，例如：
- 当前登录用户
- token
- 主题模式
- 权限信息
- 全局菜单状态

### 二类：页面级状态
适合放在页面组件内部，例如：
- 当前表单输入值
- 当前弹窗是否打开
- 当前页面筛选条件
- 当前页面局部 loading

### 三类：临时计算状态
适合通过 `computed` 派生，不建议额外存储，例如：
- 是否已登录
- 用户名显示文案
- token 截断展示内容

## 当前推荐原则
- 能由已有状态推导出来的值，不重复存储
- 跨页面共享的数据进入 Pinia
- 页面独有状态留在页面组件
- 本地存储只保存必要信息，不保存过多衍生数据

## 当前用户状态模型
当前用户状态建议至少包含：
- `token`
- `profile`
- `username`
- `roles`
- `loading`
- `refreshing`

当前已验证流程：
1. 登录成功后保存 token
2. 更新 store 中的用户状态
3. 刷新当前用户信息
4. 页面通过 store 自动响应更新

## 持久化策略建议

### 适合持久化的数据
- token
- 基础用户缓存
- 主题选择

### 不适合持久化的数据
- 瞬时 loading
- 错误提示文案
- 页面临时筛选项

原则：
- 持久化数据尽量小而稳定
- 避免把短生命周期状态写入本地存储

## Store 设计建议

### 一个 store 应聚焦一个领域
例如：
- 用户 store
- 主题 store
- 权限 store
- 菜单 store

### Store 内建议包含
- `state`
- `getters` 或 `computed`
- `actions`

### Action 职责
适合放在 action 中的内容：
- 登录
- 登出
- 刷新用户资料
- 同步本地缓存

不建议放在 action 中的内容：
- 与具体页面强绑定的 UI 表现逻辑

## 当前项目中的推荐数据来源顺序
1. 以 Pinia store 为页面直接数据源
2. store 从本地缓存做初始化恢复
3. store 再通过接口刷新最新数据

这样做的好处：
- 页面感知统一
- 刷新后体验更平滑
- 数据来源更可控

## 常见反模式
- 页面和 store 同时维护一份相同用户数据
- 只保存 token，不刷新用户资料
- 把所有 loading 都塞进全局 store
- 页面直接改本地存储，不经过 store

## 后续扩展建议
- 增加主题 store
- 增加权限 store
- 增加菜单与标签页状态管理
- 增加统一的 store 初始化入口

## 什么时候要拆更多 store
出现以下情况时应继续拆分：
- 单个 store 体积明显变大
- 状态字段开始跨多个业务域
- action 数量过多且职责混杂
- 多个页面开始依赖不同类型的全局状态
