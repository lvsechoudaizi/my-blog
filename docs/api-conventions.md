# 接口约定草稿

## 文档目的
这份文档用于沉淀当前项目的前后端接口命名、路径设计、返回结构和联调约定，减少后续接口扩展时风格不统一的问题。

## 路径约定

### 网关统一前缀
当前通过网关暴露的接口按业务域划分前缀：
- `/api/auth/**`
- `/api/blog/**`

原则：
- 前端统一调用网关地址
- 不直接绕过网关访问下游微服务

## 服务职责约定

### `/api/auth/**`
适合放：
- 登录
- 当前用户信息
- 角色权限相关接口
- 后续 token 刷新接口

### `/api/blog/**`
适合放：
- 博客管理
- 项目管理
- 示例业务接口

## 接口命名建议
- 获取列表：`GET /resources`
- 获取详情：`GET /resources/{id}`
- 新增资源：`POST /resources`
- 更新资源：`PUT /resources/{id}`
- 删除资源：`DELETE /resources/{id}`

当前阶段可以不强求完全 REST 化，但至少要保持：
- 路径语义清晰
- 动作命名统一
- 相同类型接口风格一致

## 返回结构约定
建议所有接口统一返回结构，例如：
- 状态码
- 提示信息
- 业务数据

如果项目已经在 `common` 中定义统一响应对象，则优先复用，不要每个服务各自返回不同格式。

## 请求头约定

### 认证相关头
- `Authorization`: 前端携带 token

### 用户上下文头
由网关透传：
- `X-User-Name`
- `X-User-Roles`
- `X-User-Permissions`

约定：
- 前端只负责带 token
- 用户上下文头不由前端直接传
- 下游服务优先从网关透传头读取用户信息

## 前端接口封装约定
- Axios 统一在 `src/api/http.ts` 配置
- 业务接口按域拆分，例如 `auth.ts`
- 不在页面组件里直接拼接大量请求逻辑

## 错误处理约定
- 网关鉴权失败要有统一表现
- 登录失败要返回清晰可展示的错误信息
- 前端页面只处理展示，不重复实现底层错误解析

## 联调约定
- 联调优先从登录链路开始
- 新接口接入前先确认是否走网关
- 如果下游接口依赖用户信息，先确认网关透传头是否已就绪

## 参数设计建议
- 查询条件尽量用 query 参数
- 新增和更新尽量使用请求体
- 标识资源优先使用路径参数
- 避免出现同类接口参数风格混乱

## 命名建议
- DTO 命名清楚表达用途
- 请求对象统一用 `Request` 后缀
- 响应对象统一用 `Response` 后缀
- 不要使用过于模糊的命名，例如 `Data`、`Info`

## 版本和扩展建议
当前阶段可以先不引入版本号，但后续若接口规模扩大，可考虑：
- `/api/v1/auth/**`
- `/api/v1/blog/**`
