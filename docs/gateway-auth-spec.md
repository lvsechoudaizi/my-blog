# 网关鉴权规范草稿

## 文档目的
这份文档用于明确当前 `gateway` 模块在整个系统中的鉴权职责、放行规则、用户上下文透传方式和后续扩展原则，避免认证逻辑散落到各个服务中。

## 当前定位
`gateway` 是统一流量入口，当前主要负责：
- 请求路由转发
- JWT 校验
- 公开接口放行
- 用户上下文透传

## 当前鉴权链路
1. 前端携带 token 发请求
2. 请求进入 `gateway`
3. `gateway` 判断当前路径是否属于公开接口
4. 如果不是公开接口，则校验 token
5. 校验成功后解析用户名和角色
6. 通过请求头透传给下游服务
7. 校验失败则直接拦截请求

## 公开接口约定
当前或后续通常应放行的接口包括：
- 登录接口
- 健康检查接口
- 某些公开访问页面对应的后端接口

原则：
- 放行规则应集中维护
- 不要把放行逻辑散落到多个类中
- 新增公开接口时优先更新网关规则

## Token 处理约定

### 前端职责
- 负责携带 token
- 不负责解析并生成用户上下文头

### 网关职责
- 提取 token
- 校验 token
- 解析用户身份信息
- 构造透传头

### 下游服务职责
- 使用网关透传的用户信息
- 不信任前端自定义的同名用户头

## 用户上下文透传约定
当前已透传的头包括：
- `X-User-Name`
- `X-User-Roles`

原则：
- 所有用户上下文头由网关生成
- 下游服务通过统一头获取用户身份
- 后续新增用户上下文字段时，优先沿用同一透传模式

## 当前推荐设计原则
- 鉴权逻辑优先集中在网关
- 公共 token 解析能力可复用 `common`
- 业务服务不重复解 token
- 网关拦截非法请求，减少下游服务负担

## 异常处理建议

### token 缺失
建议：
- 直接返回未授权响应
- 不进入下游服务

### token 无效或过期
建议：
- 统一返回可识别的错误状态
- 保持错误结构一致，便于前端统一处理

### 解析异常
建议：
- 网关记录必要日志
- 响应不要泄露敏感信息

## 当前实现重点
当前已做的关键点包括：
- 校验 JWT 是否有效
- 从 JWT 中解析用户名和角色
- 通过请求头透传用户信息
- 支持 `auth-center` 与 `blog-service` 的联调场景

## 新增接口时的检查项
- 是否需要鉴权
- 是否应列入公开接口
- 是否依赖用户上下文
- 下游是否已按约定读取透传头

## 常见问题

### 下游拿不到用户信息
优先检查：
- 网关是否成功解析 token
- 网关是否已写入透传头
- 下游读取的头名是否一致

### 登录接口被误拦截
优先检查：
- 放行规则是否包含登录路径
- 路由匹配是否正确

### 前端带了 token 仍返回未授权
优先检查：
- 请求头格式是否正确
- token 是否过期
- 网关使用的密钥是否与签发一致

## 后续扩展建议
- 增加刷新 token 机制
- 增加更细粒度的角色或权限透传
- 增加统一错误码
- 增加审计日志和请求链路追踪
