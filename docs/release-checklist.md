# 发布检查清单草稿

## 文档目的
这份文档用于在功能迭代后提供一份最小可执行的发布前检查清单，帮助在提交、联调和后续上线前提前发现明显问题。

## 适用场景
- 新功能开发完成后
- 结构性重构后
- 联调前
- 准备发布测试版本前

## 一、代码层检查
- 关键改动文件是否已经自查
- 是否误改了无关文件
- 是否引入了临时调试代码
- 命名、结构和目录是否符合当前项目规范

## 二、前端检查
- `npm install` 依赖正常
- `npm run build` 通过
- 登录页、首页、404 页能正常访问
- Pinia 登录态恢复正常
- 样式改动没有明显页面回归
- 新增页面是否已接入路由

## 三、后端检查
- 使用的 JDK 版本正确
- 相关服务能正常启动
- `auth-center`、`blog-service`、`gateway` 无明显启动异常
- 新增接口路径是否符合网关约定
- 公共逻辑是否应抽入 `common`

## 四、基础设施检查
- Docker Desktop 正常
- `docker compose up -d` 可正常启动基础设施
- MySQL、Redis、Nacos 状态正常
- `http://localhost:8848/nacos` 可访问
- 没有误删数据卷

## 五、联调检查
- 登录接口正常返回 token
- 前端请求能通过网关访问后端
- 网关能正确透传用户头
- 下游服务能读取当前用户信息
- 关键接口无明显 401、403、500

## 六、文档检查
- 新增规则是否已补到 `docs`
- 新增可复用流程是否已补到 `SKILL.md`
- 如果修复了真实问题，是否已补到排障文档

## 七、样式和交互检查
- 样式是否仍符合 `src/styles` 目录规范
- 页面级样式是否已拆到对应 `less` 文件
- 没有把大量样式重新堆回单文件
- 交互按钮和错误提示是否清晰

## 八、提交前检查
- 提交内容是否聚焦同一主题
- 是否存在应拆开的改动
- 是否需要补充变更说明
- 是否需要先让他人验证关键链路

## 当前最小验证路径
推荐至少走一次以下链路：
1. 启动 Docker 基础设施
2. 启动 `auth-center`
3. 启动 `blog-service`
4. 启动 `gateway`
5. 启动前端
6. 登录
7. 进入首页查看用户信息

## 高风险改动提醒
以下改动发布前应提高检查强度：
- 鉴权逻辑改动
- 网关路由改动
- Docker Compose 改动
- 样式体系重构
- 状态管理结构调整
