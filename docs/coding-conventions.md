# 编码约定草稿

## 文档目的
这份文档用于统一当前项目在前端、后端和文档层面的基础编码习惯，减少“能跑但风格越来越散”的问题。

## 总体原则
- 优先清晰，再追求技巧
- 优先一致，再追求个性
- 优先小步拆分，再避免一次性堆太多逻辑
- 优先复用已有约定，不轻易另起一套风格

## 前端约定

### 目录和文件
- 页面放在 `src/views`
- 布局放在 `src/layouts`
- 状态放在 `src/stores`
- 请求封装放在 `src/api`
- 工具函数放在 `src/utils`
- 样式统一进入 `src/styles`

### 组件命名
- 页面组件使用 `xxxView`
- 布局组件使用有语义的名称，例如 `AppLayout`
- 不使用含糊命名，例如 `Test.vue`、`Temp.vue`

### 样式约定
- 全局样式从 `src/styles/index.less` 进入
- 页面级样式单独拆分到 `styles/pages`
- 布局级样式拆分到 `styles/layouts`
- 公共样式能力尽量抽到变量、主题和 mixin

### 状态管理约定
- 跨页面共享状态进入 Pinia
- 页面私有状态留在页面组件
- 不要在页面和 store 之间维护两份相同状态

### 请求处理约定
- Axios 实例统一封装
- 页面不直接写重复的请求底层逻辑
- 请求错误优先在统一层处理，再在页面做展示

## 后端约定

### 模块职责
- 共享能力优先进入 `common`
- 认证相关优先进入 `auth-center`
- 业务接口进入业务服务
- 网关只做网关应做的路由和鉴权工作

### 类和对象命名
- 请求参数对象使用 `Request`
- 响应对象使用 `Response`
- 工具类使用清晰语义名称，例如 `JwtUtils`
- 不使用含义不明的命名，例如 `Manager2`、`DataObj`

### 配置约定
- 注册中心和配置中心相关内容优先放 `bootstrap.yml`
- 应用运行配置优先放 `application.yml`

### 鉴权约定
- token 解析优先在网关集中完成
- 用户上下文通过统一请求头透传
- 下游服务不重复解 token

## 文档约定
- 新增结构性决策尽量补到 `docs`
- 新增可复用流程尽量补到 `SKILL.md`
- 新增真实问题和解决方法尽量补到排障文档

## 命名约定
- 名称要体现职责
- 避免无意义缩写
- 同类事物尽量保持相同后缀和命名模式

例如：
- `LoginRequest`
- `LoginResponse`
- `AuthController`
- `UserStore`

## 拆分约定
- 文件变大时优先拆分，而不是继续叠加
- 页面变复杂时优先拆出业务区块
- 公共逻辑出现第二次复用时就考虑抽取

## 注释约定
- 注释用于解释“为什么”
- 不要用注释重复代码表面含义
- 临时说明性注释在问题解决后及时清理

## 提交前自检约定
- 是否改动了无关文件
- 是否保留了调试代码
- 是否破坏了目录结构一致性
- 是否补充了必要文档

## 当前阶段最重要的执行点
- 保持前后端目录分层清晰
- 保持网关鉴权入口统一
- 保持样式体系不回退到单文件堆积
- 保持状态管理边界清楚
