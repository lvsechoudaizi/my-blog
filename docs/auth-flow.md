# 认证链路说明草稿

## 文档目的
这份文档用于说明当前项目从前端登录到后端鉴权、再到下游服务获取用户上下文的完整链路，便于后续继续扩展权限系统时保持一致理解。

## 当前链路概览
当前认证链路分为四段：
1. 前端发起登录请求
2. `auth-center` 校验身份并签发 JWT
3. 前端后续请求携带 JWT 访问 `gateway`
4. `gateway` 校验并把用户信息透传给下游服务

## 详细流程

### 1. 前端登录
前端登录页收集用户名和密码，通过 Axios 请求网关暴露的认证接口。

当前关注点：
- 请求地址统一走网关
- 登录成功后保存 token
- 登录完成后刷新当前用户信息（`GET /api/auth/me`）

### 2. 网关转发到 `auth-center`
登录接口由网关转发到认证中心。

网关在这个阶段的职责：
- 路由转发
- 对公开登录接口放行

### 3. `auth-center` 签发 JWT
`auth-center` 接收登录请求后：
- 校验用户名和密码
- 组织用户信息和角色信息
- 生成 JWT
- 返回前端

当前 JWT 中至少包含：
- 用户名
- 角色信息
- 权限点信息

## 登录成功后的前端行为
前端在拿到 JWT 后需要完成：
- 保存 token
- 更新登录态
- 跳转首页
- 刷新当前用户资料

为什么需要刷新用户信息：
- 页面展示依赖用户上下文
- 只存 token 不足以支撑完整页面渲染

## 后续请求链路

### 1. 前端携带 token 发请求
前端通过 Axios 拦截器把 token 放到请求头中。

### 2. `gateway` 进行 JWT 校验
网关收到请求后：
- 提取 token
- 校验 token 是否有效
- 无效则直接拦截
- 有效则继续向下游转发

### 3. `gateway` 解析用户信息
网关从 JWT 中解析：
- 用户名
- 角色
- 权限点

然后通过请求头透传给下游服务。

当前已透传的头：
- `X-User-Name`
- `X-User-Roles`
- `X-User-Permissions`

### 4. `blog-service` 读取用户上下文
业务服务不直接重复解 token，而是优先读取网关透传的用户信息头。

这样做的好处：
- 鉴权逻辑集中
- 业务服务更轻
- 联调更容易定位问题

## 当前职责分工

### 前端
- 发起登录
- 保存 token
- 自动携带 token
- 维护用户登录态

### `auth-center`
- 负责认证
- 负责令牌签发

### `gateway`
- 负责统一鉴权
- 负责透传用户上下文

### 下游服务
- 负责业务处理
- 读取网关传递的用户信息

## 当前链路中的关键约定
- 登录接口走网关，不直接绕过网关调用服务
- 用户上下文头由网关生成
- 下游服务不信任前端自带的同名头
- token 解析逻辑集中在网关或公共模块

## 常见问题定位思路

### 登录失败
优先检查：
- 前端请求地址是否正确
- 网关是否正确转发到 `auth-center`
- `auth-center` 是否正常启动

### 登录成功但首页无用户信息
优先检查：
- 前端是否保存 token
- 前端是否调用了刷新用户信息逻辑
- 网关是否透传了用户头
- 下游服务是否正确读取请求头

### 请求被网关拦截
优先检查：
- token 是否为空
- token 是否过期
- 网关 JWT 密钥是否匹配
