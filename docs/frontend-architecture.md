# 前端架构草稿

## 文档目的
这份文档用于沉淀当前前端工程的模块划分、职责边界和演进方向，避免随着页面和功能增加，前端结构再次回到“所有东西堆在一起”的状态。

## 当前技术栈
- Vue3
- Vite
- TypeScript
- Vue Router
- Pinia
- Axios
- Less

## 当前目录结构

### `src/api`
职责：
- 封装请求实例
- 统一处理请求拦截和响应拦截
- 按业务域拆分接口文件

原则：
- 页面组件不直接处理底层请求细节
- 通用请求能力统一沉淀在这里

### `src/router`
职责：
- 路由定义
- 登录拦截
- 页面跳转规则

原则：
- 路由守卫统一放在路由层
- 登录态判断尽量不要散落到每个页面

### `src/stores`
职责：
- 维护跨页面共享状态
- 当前主要用于用户登录态和用户信息

原则：
- 共享状态进入 Pinia
- 页面私有状态留在组件内部

### `src/layouts`
职责：
- 放置页面壳层和公共布局
- 当前包含后台基础布局

原则：
- 布局组件负责结构和导航
- 页面组件负责具体业务内容

### `src/views`
职责：
- 放置页面级组件
- 当前包括登录页、首页、404 页

原则：
- 一页一个组件
- 页面组件聚焦页面行为和数据组合

### `src/utils`
职责：
- 存放与业务无强绑定的工具能力
- 当前包括 token 和登录态缓存工具

### `src/styles`
职责：
- 管理全局和页面级样式规范
- 统一采用 Less 目录化方案

## 当前分层理解
可以把前端按四层理解：
1. 展示层：`views`、`layouts`
2. 状态层：`stores`
3. 接口层：`api`
4. 基础层：`utils`、`styles`、路由配置

## 当前推荐数据流
1. 页面触发用户操作
2. 调用 `store` 或 `api`
3. 请求通过 Axios 实例发出
4. 响应结果更新 `store` 或页面状态
5. 页面根据状态重新渲染

## 当前推荐页面开发方式

### 页面职责
页面负责：
- 组织界面结构
- 触发动作
- 展示结果

页面不建议负责：
- 大量底层请求封装
- 通用状态缓存策略
- 全局鉴权规则

### 组件拆分建议
当页面开始变复杂时，优先按下面方式拆分：
- 页面容器组件
- 业务区块组件
- 通用展示组件

## 当前架构中的关键约定
- 登录态以 Pinia 为中心管理
- token 存储通过 `utils` 统一处理
- 请求入口统一由 Axios 实例处理
- 路由访问控制统一由 `router` 守卫处理
- 样式统一进入 `src/styles` 体系

## 当前已验证的做法
- 登录页和首页通过路由分离
- 登录成功后跳转首页
- 登录态支持持久化
- 刷新后可自动恢复用户状态
- 样式已完成目录化拆分

## 后续演进建议
- 增加 `components/business` 和 `components/ui` 分层
- 增加主题切换 store
- 为请求封装统一的错误提示策略
- 增加菜单、面包屑和权限导航能力
- 增加页面级接口模型定义

## 什么时候要重构前端结构
出现以下信号时，说明要继续做结构升级：
- 页面里接口逻辑越来越多
- 多个页面重复维护相同状态
- 样式文件开始跨页面互相影响
- 路由守卫逻辑越来越重
- 页面组件体积明显膨胀
